Architecture

把 API、存储、队列和控制台放到一张图里理解。

这页不是画炫酷框图，而是解释 ShiTi 当前线上最重要的几个技术层次，方便前端、后端和运维对齐职责边界。

Surface

操作面

公开入口页负责说明系统能力，`/admin` 负责租户级运营和内容管理，`/docs` 提供 OpenAPI 文档。

  • 公开前台用于说明系统和引导路径
  • 运维控制台用于高频管理动作
  • Swagger UI 用于脚本和系统集成
API Layer

NestJS 控制层

认证、题目、文档、资源、审计、导出等能力都由同一套 NestJS API 暴露，并统一错误包、请求 ID、校验和 RBAC。

  • ValidationPipe 和 DTO 负责入口收口
  • 统一错误信封负责请求语义一致
  • 租户角色控制高风险写接口
Data Plane

Postgres + Prisma + RLS

题目、taxonomy、文档、导出、资产元数据和审计日志都存于 Postgres，Prisma 负责 ORM，RLS 负责租户边界兜底。

  • 应用层校验和 RLS 双重租户隔离
  • Prisma schema 与 RLS 规则保持同步
  • 审计日志已持久化，不再依赖进程内存
Queue & Storage

Redis、Worker 与 MinIO

Redis 同时服务限流和导出队列，MinIO 承接资产和导出结果对象存储，worker 负责异步 PDF 生成。

  • 限流优先走 Redis，异常时回退内存 bucket
  • 导出队列支持取消、重试、清理
  • MinIO 失败路径已经补了回滚和 503 语义

Request Path

典型请求链路

1. Operator

在 `/admin` 或业务脚本中触发请求。

→
2. API

完成校验、RBAC、租户上下文和审计。

→
3. Storage

命中 Postgres、Redis、MinIO 或 worker。

→
4. Signals

产生日志、审计、metrics 和状态更新。
